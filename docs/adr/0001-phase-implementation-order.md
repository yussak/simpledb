# 0001. Phase の実装順序

- ステータス: 採用
- 日付: 2026-06-27

## コンテキスト

学習目的で RDBMS をゼロから自作するにあたり、対応する機能をどの順番で実装するかを決める必要がある。実装順序の案を複数比較した。

## 決定

「動くものを最短で作り、そこに肉付けする」方針のもと、依存関係が素直になる順に Phase を並べる。

| Phase | 内容 |
|---|---|
| 0 | 字句解析・構文解析（CREATE/INSERT/SELECT のみ）＋ メモリ上テーブル ＋ 素朴な実行器 |
| 1 | WHERE（フルスキャン＋フィルタ） |
| 2 | ファイル永続化（ページ単位のストレージ） |
| 3 | B+木インデックス ＋ 主キー検索 |
| 4 | WAL ＋ トランザクション |
| 5 | TCP 接続（簡易な独自プロトコル） |

主な判断:

- パーサ・メモリテーブル・実行器は Phase 0 にまとめる。3 つ揃って初めて「SQL を打つと結果が返る」動くものになるため。
- 主キー検索は B+木の後（Phase 3）に置く。効率的なルックアップはインデックスが前提のため。WHERE は先にフルスキャンで実装する。
- ファイル永続化（Phase 2）をインデックス（Phase 3）より先にする。B+木はディスク上のページ構造として実装するのが自然なため。

## 検討した代替案

比較した案の一つは、機能を次の並びで実装するものだった。

1. CREATE/INSERT/SELECT だけ対応
2. メモリ上のテーブル実装
3. SQL 字句解析・構文解析
4. ファイル永続化
5. WHERE と主キー検索
6. B+木インデックス
7. WAL・トランザクション
8. TCP 接続対応

次の難点があるため採らなかった。

- 1〜3（CREATE 等の対応・メモリテーブル・SQL 解析）が別々の Phase に分かれている。これらは単体では動作確認できず、3 つ揃って初めて「動くもの」になるため、Phase の区切りとして中途半端になる。
- 「主キー検索」（5）が B+木インデックス（6）より前にある。効率的なルックアップはインデックスを前提とするため、この順序では先に作った主キー検索をインデックス導入後に作り直すことになる。

## 結果

- 各 Phase が前の Phase の上に素直に乗り、小さい単位で実装・検証を繰り返せる。
- 最初のマイルストーン（Phase 0）で end-to-end に 1 本通るため、早期に達成感と検証基盤が得られる。
